# Bota Apps UI

A component library and design system for building data-heavy product
interfaces: context-free primitives, schema-driven forms and detail views,
charts, and an application shell — all themeable via CSS design tokens.

Every story on this site is compiled straight from package source, so what
you see here is exactly what ships to npm.

## What's inside

- **`@bota-apps/react-ui`** — UI primitives (buttons, dialogs, tables,
  navigation…), the schema-driven `DynamicForm`/`DynamicDetail`, and charts.
  Interactive primitives wrap Radix UI, so focus management, ARIA wiring, and
  keyboard behavior come from a battle-tested base.
- **`@bota-apps/react-components`** — the app-runtime layer: providers,
  `AppShell`, pages, routing, and navigation.
- **`@bota-apps/tailwind-preset`** — the Tailwind preset and theme CSS
  variables (light and dark) every component keys off of.

## How to explore

- **Controls** — every component's props are enumerated as live controls
  (variant unions become radio/select inputs). Tweak them on any story; when
  running Storybook locally you can even save a tweaked state as a brand-new
  story from the Controls panel.
- **Theme toggle** — the toolbar switch flips the design tokens between light
  and dark, so any story can be audited in both schemes.
- **Accessibility** — the a11y panel runs axe checks against the rendered
  story and reports violations, per story.
- **Docs** — each component has an autogenerated docs page with prop tables
  (descriptions and defaults straight from the source JSDoc) and copyable
  code snippets for every story.
- **Interactions** — behavior-heavy stories ship play functions; step through
  them in the Interactions panel.

## Design principles

- **Container-scoped responsiveness** — components adapt to the space they're
  given, not the viewport, so they compose into any layout.
- **Tokens over overrides** — consumers style through variants and theme
  tokens, never ad-hoc class overrides.
- **Schema-driven surfaces** — forms and detail views are generated from
  typed schemas, keeping behavior consistent across an entire app.
